# User Acceptance Testing Notes

**Date:** 2026-04-26  
**Tester:** Aryan Banerjee  
**Participants:** 6 Purdue intramural cricket teammates  
**Test Focus:** Frontend usability, event flow, and RSVP intuitiveness  

## Setup

I tested the Schedulite frontend with 6 of my intramural cricket teammates at Purdue. I chose them because they are close to the target users for our app. They already deal with practice times, game availability, group chat spam, and people not replying on time.

The test was informal. I did not give them a full walkthrough at first because I wanted to see what they would click naturally. I mainly watched them use the app and asked them to say what they thought each page or button was for.

## Notes While Testing

Most people understood the general idea pretty quickly. Once they saw groups and events, they connected it to team scheduling without needing much explanation. A few of them immediately compared it to WhatsApp polls or just texting in the group chat.

The group flow made sense after they saw a group already created. Creating a group was okay, but a couple people paused for a second because they were not fully sure what would happen after creation. They expected either a join code or some obvious next step after the group was created.

The event flow was mostly clear. They understood that an organizer would create a practice or game and then people would respond. The event title, time, and location were the pieces they looked for first. A couple people said the event card should make the RSVP status more obvious because when you are checking quickly, you do not want to hunt for whether you already replied.

For RSVP, the idea was very intuitive. Everyone understood In, Out, and Maybe. One person said Maybe should probably have a note box because sometimes for cricket it means “I can come late” or “I need a ride.” That matches what we already planned with optional notes. Another person said Out should maybe also allow a note, especially if someone is injured or has an exam.

The biggest confusion was not the concept, it was mostly small UI clarity. Some buttons could be more obvious. Some pages could use more helper text or labels. A few times they asked “am I already in this group?” or “did that event actually save?” so clearer confirmation messages would help.

The app felt more understandable when there was already sample data. With an empty account, users needed more guidance. With a sample group and event already there, they understood the purpose much faster.

## Feedback I Heard

- The main idea makes sense for cricket teams.
- RSVP is simple and easy to understand.
- In, Out, Maybe are the right choices.
- Maybe should allow a note.
- The event page should make the current RSVP status very visible.
- Group creation should have a clearer next step.
- Users want confirmation after saving an event or RSVP.
- The app should not feel like another full calendar app, it should stay quick.
- A join code or invite link needs to be easy to find.
- For actual cricket use, reminders would be useful because people forget to reply.

## Overall Result

Overall, the UAT was successful. The users understood the main purpose of Schedulite and could see how it would help a cricket team or student organization. The core workflow of group, event, and RSVP made sense.

The main issues were not major feature problems. They were mostly UI clarity issues. The app needs clearer labels, stronger confirmation after actions, and a more obvious display of RSVP status. Based on this test, the product idea is still valid, but the frontend should keep improving before being used by a larger group.

## Next Steps

- Make RSVP status more visible on event pages.
- Add clearer confirmation after group/event/RSVP actions.
- Add more helper text for empty states.
- Make join code or invite link easier to find.
- Keep testing with real student groups, not only developers.
